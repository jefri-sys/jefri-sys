## ⬡ &nbsp;Engineering Philosophy

<br />

> *Software becomes meaningful when it solves a real problem — not when it demonstrates complexity.*
> *Architecture should serve the user, not the engineer's curiosity.*

<br />

I approach every system with three questions before writing a line of code:

| | |
|:--|:--|
| **Why does this exist?** | Every feature earns its implementation by solving a named user problem. |
| **What breaks first?** | I design for failure modes before success paths. |
| **Can someone else own this?** | Code is communication. Readability is an engineering requirement. |

## ⬡ &nbsp;How I Think About Systems

<br />

```
  Input ──▶ Parse intent ──▶ Extract domain context ──▶ Resolve dependencies
                                                                │
                       Compose structured artifact ◀───────────┘
                                     │
                              LLM refines language
                                     │
                        Structured output delivered
```

<br />

I build in layers where each layer has a **single, testable responsibility**.
Logic does not belong in the LLM.
The LLM does not belong in the engine.
Data does not belong in the API layer.

Separation isn't a style preference — it's what makes a system maintainable six months later, by someone who wasn't there when it was built.

## ⬡ &nbsp;Featured Projects

<br />

<details>
<summary><b>▸ &nbsp;Synapse &nbsp;—&nbsp; AI-Powered Student Productivity Platform</b></summary>

<br />

**The Problem**

Students carry their career assets — resumes, certificates, transcripts, internship documents — in disconnected folders with no intelligence layer on top. Resume tailoring is manual and generic. Collaboration is ad-hoc. There's no system that *understands* your documents.

**What Synapse Does**

Synapse is a production-grade SaaS platform that gives students a unified career workspace: intelligent document analysis, AI-assisted resume refinement, semantic search across their own data, and real-time collaboration tools — all in one place.

**Architecture**

```
  Document Upload (PDF / Image)
           │
     OCR + Parsing Layer
           │
     Chunking + Embedding  (Gemini Embeddings API)
           │
     Vector Store  ──  MongoDB with semantic index
           │
     RAG Retrieval ──▶ Context Composition ──▶ LLM Generation
                                                  │
                                     Gemini (quality) / Groq (latency)
                                                  │
              Career Vault ── Resume Intelligence ── Job Match Engine
```

**Engineering Decisions**

- **Multi-LLM architecture** — Gemini handles quality generation; Groq handles latency-critical paths. Switching is configurable at runtime without touching application logic.
- **RAG over fine-tuning** — Student data is dynamic and personal. A static fine-tune cannot adapt to a new document upload. RAG retrieves from the actual live vault.
- **Programmatic prompt composition** — Prompts are assembled from extracted context, not hardcoded. Templates adapt to document type and user intent at request time.
- **Atomic JWT sessions** — Access + refresh token rotation via httpOnly cookies. No client-accessible tokens. No LocalStorage exposure.
- **Cloudinary pipeline** — All media is processed, optimized, and served through Cloudinary CDN. Raw blobs never hit MongoDB.

**Real-time Layer**

Socket.io for persistent messaging with presence indicators, read receipts, and typing state.
WebRTC for peer video calls with Socket.io-based signaling and clean connection teardown.

**Stack**

`React` · `Node.js` · `Express` · `MongoDB` · `Socket.io` · `WebRTC` · `Gemini` · `Groq` · `Cloudinary` · `JWT` · `PWA`

<details>
<summary><b>▸ &nbsp;SCRMS &nbsp;—&nbsp; Smart Campus Reprography Management System</b></summary>

<br />

**The Problem**

Saintgits' reprography counter ran entirely on paper and verbal coordination. Students queued physically. Staff had no demand visibility. There was no order tracking, no payment intelligence, and no analytics. A high-traffic campus service with zero operational infrastructure.

**What SCRMS Does**

SCRMS replaces the physical queue with a full-stack campus operations platform: digital order management, real-time tracking, role-based workflows, wallet payments, group orders with split billing, and an admin analytics layer — deployed as a PWA for zero-install adoption.

**Architecture**

```
  Student / Faculty / Staff / Guest
           │
     Role-based Access Control
     (Priority weights: Faculty 100 · Staff 150 · Student 200 · Guest 200)
           │
     Order Workflow
     Submit ──▶ Queue ──▶ Processing ──▶ Ready ──▶ Collected
           │
     Token System: DDMMYYYY-NNNN  (deterministic, date-sortable, collision-free)
           │
     Wallet Engine ──▶ Razorpay Top-Up ──▶ Atomic MongoDB Transactions
           │
     Real-time: Socket.io ──▶ Web Push Notifications (VAPID)
           │
     Admin Layer: Analytics · Inventory · Revenue · Demand Patterns
```

**Engineering Decisions**

- **Atomic transactions** — Wallet debits, group split payments, and refunds execute inside MongoDB multi-document transactions. Partial state is architecturally impossible.
- **Token format `DDMMYYYY-NNNN`** — Human-readable, date-sortable, and counter-based. No UUID lookup needed for staff verification.
- **Role weights, not binary flags** — Roles carry numeric priority weights that influence queue position without surfacing that complexity to users.
- **PWA + Web Push** — Students receive order-ready notifications without polling. Offline state is handled gracefully by service workers.
- **Group orders** — Multiple students pool into a single order. Wallet-to-wallet transfers are atomic. PDF receipts are generated server-side via `pdfkit` and served immediately.
- **Replica set requirement** — MongoDB Atlas was configured with a replica set. This was a hard dependency for multi-document transaction support — not an optional upgrade.

**Deployment**

Vercel (frontend) · Railway (backend) · MongoDB Atlas (replica set) · Cloudinary (`scrms-chat`)

**Stack**

`React PWA` · `Node.js` · `Express` · `MongoDB Atlas` · `Socket.io` · `Razorpay` · `Cloudinary` · `JWT` · `Web Push (VAPID)` · `pdfkit` · `Brevo SMTP`
